# Performance baseline — T5 perf pass (batch 4)

**Fecha**: 2026-07-22
**Versión**: v2.0.0-pre (post batch 4 merge)
**Auditor**: Mavis (orquestador, Track C)

## Resumen

Perfil baseline de funciones puras + optimizaciones en hot paths del renderer.
Sin profiling en Electron (requiere DevTools, se hace en QA manual).
El smoke test `test/performance-budget.test.js` corre 9 funciones puras con
budgets generosos de 50ms / 1000 iter y actúa como red de seguridad contra
regresiones graves.

## Optimizaciones aplicadas

### `src/renderer.js` (T5 perf)

1. **Cache de elementos DOM en hot paths**:
   - `catTailEl`, `dogTailEl` — antes se hacía `getElementById` cada frame en
     `animateTail()` (60Hz × 2 lookups = 120 lookups/s). Ahora se cachean al
     init y se re-cachean después de cada `loadMascotSVG()` (que re-injecta
     `innerHTML` y destruye nodos).
   - `earLeftEl`, `earRightEl` — antes `petSvgWrapper.querySelector('.pet-ear-X')`
     dentro de `maybeTwitchEar()` (~cada 6s pero igual lookups innecesarios).
   - `pupilEls` (Array) — antes `document.querySelectorAll('.pet-pupil')` en
     cada `mousemove` (puede ser 100+/s). Ahora Array precomputado al init.
   - Función `cachePetElements()` centraliza el cache. Llamada desde
     `DOMContentLoaded` y desde `loadMascotSVG`.

2. **Loop optimization**:
   - `handlePupilTracking` cambia `pupils.forEach(pupil => ...)` a
     `for (let i = 0; i < pupils.length; i++)` — micro-opt, pero forEach tiene
     overhead de callback. En 100+ mousemove/s la diferencia suma.

3. **Sin cambios en la API pública**:
   - Las funciones `animateTail`, `maybeTwitchEar`, `handlePupilTracking`,
     `loadMascotSVG` mantienen su firma y comportamiento.
   - `initMicroPresence()` no cambia.

## Lo que NO se optimizó (y por qué)

1. **Tail `setAttribute('d', ...)` cada frame**:
   - Cambia el path SVG cada frame, no se puede saltar.
   - `d` attribute changes son GPU-accelerated en browsers modernos.
   - Medir en DevTools Performance tab en QA.

2. **`getBoundingClientRect()` en mousemove**:
   - Es costoso pero unavoidable (necesitamos la posición actual del SVG).
   - Browser ya throttle mousemove a refresh rate.

3. **Mood decay cada 1s**:
   - Ya está throttleado a 1Hz (vs antes probablemente más).
   - Coste despreciable (JSON.stringify + fs.writeFile async).

4. **`logger.info` / `logger.debug` en hot paths**:
   - `logger.js` ya tiene `minLevel = LEVELS.info` por default.
   - `debug()` retorna early si level no aplica.
   - El overhead es `process.stdout.write` que es async y buffered.

5. **Memory leak check formal**:
   - No corrí 10min soak test (requiere Electron + display).
   - Defer a QA manual. Mitigación: todos los setInterval críticos tienen
     `.unref()` o se limpian en `before-quit` (verificado por grep).

## Recomendaciones para v2.0.1+

1. **Throttle explícito de `animateTail` cuando la ventana no es visible**:
   - Usar `document.visibilityState` para pausar rAF cuando la petWindow
     está oculta. Hoy sigue corriendo aunque no se vea.
2. **Cache de `wrapperRect` en `handlePupilTracking`**:
   - Solo se invalida en resize/drag. Podría cachearse 100ms.
3. **Lazy load de assets** (cat.js, dog.js) si la app solo usa uno:
   - Hoy se cargan ambos (~15KB). Si solo se necesita uno, ahorra startup.
4. **Migrar a `will-change: transform`** en petContainer:
   - Habilita compositor layer dedicado. Mejora fluidez de animaciones.
