## Issues a resolver en este batch

### MEDIUM-1: Agregar CSP al dashboard

**Por qué**: el dashboard (`src/dashboard.html`) no tiene Content-Security-Policy.
Si un día se inyecta un XSS via `innerHTML` mal usado, el atacante puede cargar
scripts externos.

**Fix propuesto** (en Track A, ya que toca `dashboard.html`):

```html
<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src 'self' https://api.minimax.chat https://api.github.com;">
```

- `default-src 'self'`: solo recursos del mismo origen
- `script-src 'self'`: solo scripts locales (no inline)
- `style-src 'self' 'unsafe-inline'`: necesario porque `dashboard.html` tiene estilos inline
- `img-src 'self' data:`: sprites locales + data URIs (para algunos assets)
- `connect-src`: API de M3 + GitHub (para auto-update checks)

**Riesgo**: podría romper algo que hoy depende de inline scripts. Hay que testear
en QA. Si rompe, ajustar `script-src` con `'unsafe-inline'` como fallback temporal
y abrir issue para refactor.

## Recomendaciones post-v2.0.0

1. **Cert EV Windows** ($300/yr) cuando la adopción justifique. Migrar a signed
   installer. Cambia 1 línea en `package.json` build config.
2. **Auto-update server dedicado** (S3 + CloudFront) si GH Releases se queda
   corto (rate limits, geo, etc). Por ahora GH Releases es suficiente.
3. **CSP estricto** completo (sin `unsafe-inline`). Requiere refactor de
   estilos inline en dashboard.
4. **SBOM** (Software Bill of Materials) generado por CI para auditoría
   enterprise. Herramienta: `cyclonedx-npm`.
5. **Notarization macOS** si en el futuro hay build para Mac.
6. **CodeQL** en CI (GH tiene esto gratis) para análisis estático continuo.
