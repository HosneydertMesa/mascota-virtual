## Resumen ejecutivo

La mascota ahora **recuerda cosas entre sesiones**. Cada mensaje del usuario pasa por un extractor IA que decide si hay un hecho memorable para guardar (max 50), se redacta PII automáticamente, y al iniciar nueva conversación los recuerdos relevantes se inyectan al system prompt via TF-IDF lite.

Antes: "Hola, ¿qué te gustaba?" → "Miau, no me acuerdo, soy una gatita" 😅
Ahora: "Hola, ¿qué te gustaba?" → "Miau Jorge! Me dijiste que te gusta el café y que trabajas como developer ¿verdad? 🐱☕"

5 commits, 14 archivos, +2116/-8 LOC, +96 tests (de 243 → 339).

## Features

### P3 — Recuerdos persistentes (Track A)

**Concepto**:
- 50 recuerdos máximo, persistidos en `<userData>/pet-memories.json`
- Cada mensaje del usuario pasa por el extractor IA (temperatura 0.2) que decide si hay info memorable
- Si hay: 0-1 recuerdos por turno, se redacta PII (ON por default), se deduplica (Jaccard 0.7), se prunea al límite
- Al iniciar nueva conversación: top 5 recuerdos relevantes se inyectan al system prompt (bag-of-words TF)

**Archivos nuevos**:
- `src/core/pet-memories.js` (254 LOC): pure functions — tokenización, TF bag-of-words, PII redaction (email/phone/CC), dedup Jaccard, format, prune, UMD-lite
- `src/services/memories-store.js` (264 LOC): persistencia versionada, escritura atómica (.tmp + rename), addMemory integra PII + dedup + prune, setRedactPII
- `src/services/memory-extractor.js` (168 LOC): extractor IA con prompt dedicado, parseExtractorResponse pure (maneja think tags, markdown, JSON embebido)

**Archivos modificados**:
- `main.js`: importa memories-store + extractor, wire en `ai:send-message` (inject top-5 al prompt + extract en background), IPC handlers (`memories:list`, `:remove`, `:clear`, `:set-redact`)
- `src/services/ai.js`: `sendMessageToMiniMax` acepta nuevo param `memoriesContext` (7mo arg)
- `preload.js`: expone `getMemories`, `removeMemory`, `clearMemories`, `setMemoryRedact`
- `src/dashboard.html`: nueva tab "Recuerdos" con lista + clear + toggle PII
- `src/dashboard-renderer.js`: `loadMemoriesList()` con DOM API (no innerHTML con user data → imposible XSS)
- `src/styles.css`: estilos del list de recuerdos
- `package.json`: agregados `pet-memories.js`, `memories-store.js`, `memory-extractor.js` al script `check`

## Decisiones de diseño

| Decisión | Por qué |
|---|---|
| TF bag-of-words sin IDF | Corpus chico (max 50), IDF aporta poco. Si crece, se cambia. |
| PII redaction ON por default | Privacy first. Toggle off opt-in en dashboard. |
| Temperature 0.2 en extractor | Consistencia > creatividad al extraer. |
| Max 1 recuerdo por turno | Evita spam. Si hay más, el usuario los agrega manualmente. |
| Jaccard 0.7 para dedup | Balance entre "no duplicar reformulaciones" y "no rechazar info nueva". |
| Extracción en background (no bloquea reply) | UX: el reply del chat se muestra rápido, el recuerdo se guarda después. |
| Escritura atómica (.tmp + rename) | Si se corta la energía, no queda JSON corrupto. |
| DOM API en dashboard (no innerHTML) | Imposible XSS via texto de recuerdo. |
| Memorias globales (no per-pet) | La mascota "conoce al usuario" en general. Per-pet en v2 si hay feedback. |
| `memories:list` con `isKnownSender` (no isDashboardSender) | Read es seguro desde cualquier ventana, mutate requiere dashboard. |

## Limitaciones conocidas

1. **Extractor consume API quota**: 1 llamada extra por mensaje (~200 tokens, ~4000/día para usuario activo).
2. **PII redaction falsos positivos**: phone regex permisivo puede matchear secuencias 4+4+4. Conservative es mejor (false positive > false negative).
3. **Memorias globales, no per-pet**: si el usuario tiene Luna + Max, comparten recuerdos.
4. **No retry on extract fail**: si la API falla, el recuerdo se pierde. Best-effort.
5. **No multi-window**: solo se probó 1 dashboard + 1 pet window.

---

## Seguridad

- `isDashboardSender` para todas las mutaciones de memories (remove/clear/set-redact). El pet window no puede modificar recuerdos.
- PII redaction antes de persistir — el JSON en disco solo tiene texto redactado.
- No XSS en UI (DOM API, no innerHTML con user data).
- Atomic write previene corrupción del archivo.
