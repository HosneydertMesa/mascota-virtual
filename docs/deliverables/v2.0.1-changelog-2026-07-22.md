## 🐛 Bug fix crítico

### Silent mode, calendar .ics, briefing diario, pet name — no se cargaban en Settings

**Reportado por**: @HosneydertMesa, 2026-07-22, ~2h después del release v2.0.0
**Síntoma**: "Abrí las config y solo muestra pomodoro, pero el resto no sirve mal"
**Versiones afectadas**: v2.0.0 (sólo)
**Versión fixeada**: v2.0.1

#### Root cause

`initSettings()` en `src/dashboard-renderer.js` (función que se llama al abrir
el dashboard) tenía 2 awaits sin try/catch:

```js
await refreshStats();          // línea 308
setInterval(refreshStats, 60_000);
await refreshMoodWidget();     // línea 312
setInterval(refreshMoodWidget, 5000);
```

`refreshStats()` y `refreshMoodWidget()` tienen su propio try/catch interno
**pero solo wrappean el IPC**, no el resto del cuerpo. Si `window.PetMoodLabels`
o `window.PetMood` no estaban disponibles (por race condition de scripts, CSP,
o cualquier otra razón), `refreshMoodWidget` tiraba un `TypeError: Cannot
read properties of undefined (reading 'MOOD_LABELS')` que se propagaba a
`initSettings` y la cortaba.

Como el resto de la configuración de Settings (silent mode, calendar .ics,
briefing diario, pet name) se cargaba **DESPUÉS** de esos 2 awaits, nunca
se ejecutaba. El usuario veía todos los inputs/checkboxes en la UI (estaban
en el HTML), pero sin listeners y sin valores cargados → parecían "no
funcionar".

El único bloque que se cargaba bien era el de **Pomodoro** (template dropdown,
custom inputs, stats) porque estaba antes de los awaits problemáticos.

#### Fix (2 partes)

**Parte 1**: try/catch alrededor de los 2 awaits en `initSettings`:

```js
try {
  await refreshStats();
} catch (e) {
  console.error('refreshStats fallo en init:', e);
}
setInterval(refreshStats, 60_000);

try {
  await refreshMoodWidget();
} catch (e) {
  console.error('refreshMoodWidget fallo en init:', e);
}
setInterval(refreshMoodWidget, 5000);
```

**Parte 2**: defensive guards dentro de `refreshMoodWidget` (defense in depth):

```js
if (!window.PetMood || typeof window.PetMood.deriveState !== 'function') return;
if (!window.PetMoodLabels || typeof window.PetMoodLabels.MOOD_LABELS !== 'object') return;
if (typeof window.PetMoodLabels.getMoodLabel !== 'function') return;
```

Esto evita que `refreshMoodWidget` tire incluso si las dependencias de UI
no están cargadas — sale silencioso, el mood widget muestra defaults, y
el resto del dashboard sigue funcionando.

#### Lo que NO cambió

- API pública de las funciones (mismas signatures)
- Tests (818/818 siguen pasando)
- Configuración de build, dependencias, IPC handlers
- Stores JSON, persistencia

#### Lecciones aprendidas

1. **Un `await` sin try/catch en una función async mata toda la cadena**.
   Si un await puede tirar (aunque sea por una condición rara de DOM/UI),
   hay que wrappear.
2. **Defensive coding en init flows** — si una parte falla, el resto debe
   seguir intentando. Los UI widgets son nice-to-have, pero el wire de
   los settings (silent mode, calendar) es crítico.
3. **Manual QA deferred = bugs en producción**. El plan decía "QA manual
   del .exe deferred al user". El user probó en 2h y encontró el bug.
   Mitigación futura: en lugar de deferir todo, hacer al menos un smoke
   test automatizado que simule `initSettings` con mocks de `window` y
   `document`.
