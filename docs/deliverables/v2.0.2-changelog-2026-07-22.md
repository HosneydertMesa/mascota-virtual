# v2.0.2 — Changelog (HOTFIX)

**Fecha**: 2026-07-22

## Resumen

v2.0.2 arregla un bug crítico que dejaba la app completamente inutilizable: el dashboard renderizaba HTML pero NINGÚN botón respondía (ni el X cerrar). Bug latente desde v1.6.0 que se activó en v2.0.0/v2.0.1.

## 🐛 Bug fix

### Strict mode scope conflict entre `context-awareness.js` y `dashboard-renderer.js`

**Síntoma**:
- Dashboard HTML se renderiza (Pomodoro tab visible)
- Ningún botón responde, ni cerrar
- Pet se ve detrás del dashboard (window ordering, secundario)
- DevTools console: `Uncaught SyntaxError: Identifier 'shouldEnterDoNotDisturb' has already been declared (at dashboard-renderer.js:1)`
- Se reproduce tanto en `npm start` como en `MascotaVirtual-Setup-2.0.x.exe`

**Root cause**:
`src/core/context-awareness.js` declaraba `function shouldEnterDoNotDisturb(...)` a top-level (global scope del dashboard window). `src/dashboard-renderer.js` (línea 799) hacía `const { shouldEnterDoNotDisturb, ... } = window.ContextAwareness;` también a top-level. En strict mode (Annex B), tener `function` y `const` con el mismo identificador en el mismo scope es SyntaxError (lexical declaration collision).

**Fix**: Wrappar `context-awareness.js` en IIFE. Las funciones quedan en scope privado del IIFE y NO contaminan el global. La API pública se sigue exponiendo via `window.ContextAwareness` (browser) y `module.exports` (Node). Mismo patrón que ya usaba `pet-protocol.js`.

```js
// Antes (v2.0.1) — top-level functions contaminan global
function shouldEnterDoNotDisturb(keystrokes, ...) { ... }
function shouldExitDoNotDisturb(keystrokes, ...) { ... }
// ...
if (typeof window !== 'undefined') window.ContextAwareness = { ... };

// Después (v2.0.2) — IIFE wrapper, funciones privadas
(function (root) {
  function shouldEnterDoNotDisturb(keystrokes, ...) { ... }
  function shouldExitDoNotDisturb(keystrokes, ...) { ... }
  // ...
  if (typeof module !== 'undefined' && module.exports) {
    module.exports = ContextAwareness;
  } else {
    root.ContextAwareness = ContextAwareness;
  }
})(typeof window !== 'undefined' ? window : globalThis);
```

**Por qué el bug se activó en v2.0.0 y no antes**:
El bug existía latente desde v1.6.0 (commit `0dbaeb1` que agregó context-awareness.js). La regla strict mode rechaza el solapamiento desde siempre, pero V8 históricamente era más permisivo con function declarations a top-level. La reorganización de archivos en batch 4 (v2.0.0) probablemente cambió el orden de parsing y expuso el conflicto. v2.0.1 (hotfix) NO tocaba este archivo, así que el bug persistió.

## 🔮 Post-mortem del regression gap

**Por qué v2.0.0 → v2.0.1 → v2.0.2 necesitó dos hotfixes seguidos**:

1. v2.0.0 introdujo el bug de `initSettings` (await sin try/catch)
2. v2.0.1 (HOTFIX 1) arregló `initSettings` con try/catch
3. El bug del scope conflict (este HOTFIX 2) era INDEPENDIENTE y pre-existente, pero quedó oculto detrás del bug de `initSettings`. Cuando v2.0.1 arregló `initSettings`, el scope conflict se hizo visible porque `refreshStats` ya no moría antes de llegar al destructure.

**Lección aprendida**:
- El smoke test solo valida stdout/stderr, no DevTools console. Por eso no detectó el bug.
- Los tests son `node --test` puros, no testean la cadena de scripts del browser.
- Necesitamos un test E2E del dashboard que abra la ventana, valide event listeners attachados, y cierre. Diferido a v2.0.3.

**Defensa en profundidad que agregamos**:
- IIFE wrap en `context-awareness.js` elimina la posibilidad de scope conflict
- Guard estructural (test 5) detecta cualquier regression futura
- Sentinel bidireccional (test 4) garantiza que el regression test no sea placebo
